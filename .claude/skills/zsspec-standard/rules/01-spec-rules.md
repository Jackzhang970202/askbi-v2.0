# 命名规范

**生效日期**: 2026-03-19

---

## 编号格式

| 类型 | 格式 | 示例 |
|------|------|------|
| 功能需求 | REQ-{模块路径}-{功能} | REQ-用户管理-用户新增 |
| 非功能需求 | NFR-{类型}-{子项} | NFR-性能-响应时间 |
| 任务 | TASK-{模块路径}-{功能}-{中文标识} | TASK-用户管理-用户新增-后端接口 |
| 检查项 | CHK-{模块路径}-{功能}-{中文标识} | CHK-用户管理-用户新增-核心验证 |

## 目录结构

```
specs/
├── requirement/
│   ├── 01-non-functional.md
│   └── {模块路径}/REQ-{模块路径}.md
├── design/
│   ├── 01~05 全局设计文档
│   └── {模块路径}/DES-{模块路径}-{前后端}.md
├── task/
│   └── {模块路径}/TASK-{模块路径}.md

    └── {模块路径}/CHK-{模块路径}.md
```

## 版本管理

- 文档版本: v{主}.{次}，如 v0.1
- 状态: 未开始 / 进行中 / 已完成 / 已废弃

### 何时更新文档版本号

| 场景 | 操作 | 示例 |
|------|------|------|
| 新建文档 | 设为 v0.1（草稿） | v0.1 |
| 小修改（错别字、格式、微调描述） | 不更新版本号 | 保持当前版本 |
| 功能变更（增删改需求/任务/检查项） | 次版本号 +1 | v0.2 → v0.3 |
| 重大变更（模块边界调整、方案重选） | 主版本号 +1 | v0.9 → v1.0 |

### 变更记录格式

每次版本号变更时，必须在文档末尾追加变更记录：

| 版本 | 日期 | 变更内容 | 变更人 |
|------|------|----------|--------|
| v0.2 | 2026-05-11 | 增加XXX功能需求 | git user |
| v0.1 | 2026-05-10 | 初始版本 | git user |

### 关键约束

- 功能需求的改动才会涉及版本号的变更，文档中状态、格式等的改变不处理版本号
- 禁止无实质内容变更时更新版本号
- 禁止跳过版本号（如 v0.1 直接到 v0.3）
- 变更记录必须说明"改了什么"，不得只写"更新"
- 全局非功能需求不再单独生成验收检查清单文档；相关约束需在涉及模块的 `CHK-{模块}.md` 中落地，并保持对 `01-non-functional.md` 的可追溯关系。
