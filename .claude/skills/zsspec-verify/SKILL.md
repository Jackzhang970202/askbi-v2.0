# ZSSpec Verify - 开发前一致性校验、门禁判断与全局治理检查

在进入 `zsspec-apply` 前，对当前模块或子功能的 REQ / DES / TASK / CHK 先做一致性校验，再做开工门禁判断；在需要时，也可对模块级或全局 specs 做健康治理检查。

核心原则只有一句：

> 先校验文档之间有没有硬错误，再判断这些问题会不会阻断当前开发或暴露全局治理风险。

`zsspec-verify` 的目标不是做全面审查，而是在恰当范围内发现会阻断实现、引发返工或破坏 specs 体系健康度的关键问题，并据此给出明确结论与下一步动作。

## 职责边界

`zsspec-verify` 只负责：
- 对当前子功能执行开发前一致性校验与 apply 门禁判断
- 对当前模块执行模块级 specs 完整性与一致性检查
- 对整个项目执行全局 specs 健康治理检查
- 检查 REQ / DES / TASK / CHK 之间的编号追溯、覆盖关系、设计契约、任务完整性、CHK 协议完整性、版本同步与范围内依赖关系
- 识别关键断链、关键遗漏、关键不匹配、关键未决项与全局治理风险
- 基于校验结果给出局部门禁结论或模块/全局健康结论
- 输出简洁、明确、可执行的检查报告与下一步建议

不负责：
- 直接生成或重写完整 spec 文档（由 `zsspec-spec` 负责）
- 替代 `zsspec-change` 做所有变更修订
- 直接进入开发实现（由 `zsspec-apply` 负责）
- 做 code review、security review、性能审查或大范围架构巡检
- 默认把全局 verify 扩展成代码现实审计；仅当 spec 明确依赖现有实现、现有接口、现有组件或旧逻辑兼容时，才允许轻量扩展检查
- 最终验收判定（由 `zsspec-done` 负责）

## HARD GATES

<HARD-GATE>
在完成 verify 并给出“允许进入 apply”的结论前，禁止直接进入 `zsspec-apply`。
</HARD-GATE>

<HARD-GATE>
若 REQ / DES / TASK / CHK 之间存在影响当前开发范围的关键断链、关键遗漏、关键不匹配或关键未决项，不得判定为可开工。
</HARD-GATE>

<HARD-GATE>
若用户当前诉求与现有 spec 不一致，必须先以用户当前确认的诉求为准，要求修订文档后再决定是否进入 apply。
</HARD-GATE>

<HARD-GATE>
若当前实现仍需要开发者对关键功能点、关键边界、关键接口契约、关键验收标准进行大面积猜测，不得判定为可开工。
</HARD-GATE>

<HARD-GATE>
若存在必须先向用户确认、否则会改变当前实现路径、范围边界、接口契约或验收口径的问题，不得直接进入 apply。
</HARD-GATE>

<HARD-GATE>
verify 只负责校验、判断与结论输出；若需要修订文档，必须转 `zsspec-change`、`zsspec-spec` 或 `zsspec-brain`，不得在问题未收敛前直接开始开发。
</HARD-GATE>

<HARD-GATE>
全局 verify 的默认范围仍是 specs 文档体系本身，不得无边界扩展为全仓代码审计。
</HARD-GATE>

## 使用方式

```bash
/zsspec-verify <模块>
/zsspec-verify <模块> --focus=<子功能>
/zsspec-verify <模块> --scope=subfeature
/zsspec-verify <模块> --scope=module
/zsspec-verify --scope=global
/zsspec-verify --scope=global --only=traceability
/zsspec-verify --scope=global --only=coverage
/zsspec-verify --scope=global --only=version
/zsspec-verify --scope=global --only=dependency
/zsspec-verify --scope=global --only=nfr
/zsspec-verify --scope=global --only=naming
/zsspec-verify --scope=global --mode=full
```

## Scope 定义

| scope | 含义 | 默认结论类型 | 默认目标 |
|---|---|---|---|
| `subfeature` | 当前模块中准备开工的一个小子功能 | 可开工 / 有条件可开工 / 暂不可开工 | apply 前门禁 |
| `module` | 当前模块下整体 spec 链路 | 健康 / 有风险 / 不达标 | 模块级 specs 治理 |
| `global` | 整个项目的 specs 体系 | 健康 / 有风险 / 不达标 | 全局 specs 治理 |

说明：
- 默认 `scope=subfeature`
- 若用户未指定 `scope`，必须优先按当前最接近开工的那一小块处理
- `module` / `global` 模式默认不直接替代某个子功能的 apply 门禁；若用户还要判断能否开工，应额外收敛到 `subfeature`

## 默认行为

默认按以下原则执行：
- 默认 scope：`subfeature`
- 默认范围：只检查当前模块中“本次准备开工的那一小块”
- 默认目标：先查文档之间有没有硬错误，再判断现在能不能写
- 默认输出：失败项逐条列出，警告项尽量折叠
- 默认问题重点：只突出会阻断开工或破坏 specs 健康度的关键问题
- 默认不做大范围仓库扫描、现有实现巡检或架构优化审查

若用户未指定 `--focus`，应优先收敛到“当前最接近开工的子功能”；若模块过大，不得直接扫描整个功能域，应先明确本次实际检查范围，再继续 verify。

## 进入条件

### 进入 `subfeature` verify 前，至少应满足：
- 当前模块已有 REQ / DES / TASK / CHK，或至少已生成可供校验的主要 spec 文档
- `zsspec-brain` 已完成当前模块的需求收敛，或用户已明确跳过进一步讨论
- 用户希望在开发前确认当前 spec 是否达到可执行标准
- 当前模块尚未正式进入开发，或希望在开发前补一次开发前一致性校验

### 进入 `module` verify 前，至少应满足：
- 当前模块已有主要 spec 文档
- 用户希望检查当前模块整体 specs 是否齐套、同步、可维护
- 当前检查目标是模块治理，不是立即进入某个未收敛子功能开发

### 进入 `global` verify 前，至少应满足：
- 项目已有稳定的 specs 目录结构
- 用户希望检查整个项目的 specs 体系健康度
- 当前检查目标是全局治理、范围管理或阶段性收口，不是替代局部 apply 门禁

若不满足：
- 不得给出完整 verify 结论
- 应先补齐缺失文档，或回到 `zsspec-brain` / `zsspec-spec`

## 最终目标

`zsspec-verify` 最终只回答两类问题：

### 对 `subfeature`：
1. 当前模块或子功能的 REQ / DES / TASK / CHK 之间，是否存在会影响开发的硬问题？
2. 当前这块 spec 是否已具备直接开工所需的最小充分规格？

### 对 `module` / `global`：
1. 当前模块或全局 specs 体系，是否存在结构性断链、覆盖缺口、治理失衡或同步风险？
2. 当前 specs 体系整体是否处于可维护、可演进、可支撑后续开发的健康状态？

## 两阶段执行模型

### 第一阶段：一致性校验

先做结构化、规则化的文档一致性检查，识别断链、漏覆盖、不匹配、版本不同步、齐套性缺口或依赖治理问题。

### 第二阶段：结论判断

- 对 `subfeature`：判断这些问题是否阻断当前模块或子功能进入 apply
- 对 `module` / `global`：判断这些问题是否说明当前 specs 体系处于健康、有风险或不达标状态

说明：
- 一致性校验先于结论判断
- 没有完成一致性校验，不得直接给出“可开工”或“健康”结论
- 结论必须以第一阶段的事实结果为基础，而不是只做主观审查

## 审查范围收敛规则

### `subfeature`
默认只围绕当前模块的本次开发范围检查，不扩展到整个业务域。

| 场景 | 正确做法 |
|---|---|
| 用户管理模块准备做“用户新增” | 只检查“用户新增”相关 REQ / DES / TASK / CHK |
| 导出功能准备开发 | 只检查导出功能链路，不扩展扫描整个模块 |
| 模块下有多个子能力 | 先收敛本次子功能，再执行 verify |

若当前模块过大、边界不清，必须先明确本次检查对象；若连检查范围都无法收敛，应倾向回退 `zsspec-brain`。

### `module`
默认围绕单模块整体 specs 检查，但不扩展到无关模块。

重点关注：
- 当前模块文档是否齐套
- 当前模块各子功能之间是否存在结构性断层
- 当前模块状态、版本、检查链路是否同步

### `global`
默认围绕项目 specs 体系检查，但不扩展为代码现实审计。

重点关注：
- 模块集合是否一致
- 全局 NFR 与模块 CHK 是否有效承接
- 跨模块依赖、命名规范、版本同步、孤儿文档等治理问题

## 一致性校验维度

### 1. 编号追溯

检查各文档之间的需求编号、引用关系与关联链路是否可追溯。

重点检查：
- 每个 TASK 关联的 REQ 是否存在
- 每个 CHK 关联的 REQ 是否存在
- DES 中引用的需求点是否可在 REQ 中定位
- 文档中引用的模块名、功能点名、需求编号是否一致

典型失败：
- TASK-用户管理-查询-001 引用了不存在的 REQ-用户管理-查询
- CHK 引用了已删除或未定义的需求编号
- DES 描述了 REQ 中不存在的功能点

失败处理：
- 列出断链编号或不一致项
- 若影响主链路，至少判为 `Critical`
- 若是局部但会导致误开发，至少判为 `Major`

### 2. 需求覆盖

检查 REQ 中定义的关键功能结果、验收标准与检查要求，是否在 CHK 中形成对应检查项与证据要求。

重点检查：
- 每个关键验收标准在 CHK 中是否有对应检查项
- 每个关键检查项是否能回溯到 REQ
- 每个检查项是否带有合法的 `automation=static/auto/manual` 标记
- 检查项是否说明通过证据来自截图、日志、接口响应、测试结果或其他明确证据

典型失败：
- REQ 写明“支持导出 CSV”，CHK 中没有任何导出验证项
- CHK 有检查项，但没有对应需求来源
- CHK 有检查项，但缺少 `automation` 标记或标记不合法
- CHK 有验证动作，但没有证据要求

失败处理：
- 主线验收缺失判 `Critical`
- 次级功能缺失判 `Major`
- 仅证据粒度偏弱但不影响主线判断，可判 `Warning`

### 3. 设计一致

检查 DES 中的 API 契约、状态流、数据结构、页面流程与关键设计是否与 REQ 的功能定义匹配。

重点检查：
- DES 的 API 契约是否支撑 REQ 的功能能力
- DES 的主流程是否覆盖 REQ 的成功路径与必要失败路径
- DES 的字段与数据结构是否满足 REQ 的业务语义
- DES 是否遗漏会影响当前实现的关键规则、校验或状态转换

典型失败：
- REQ 要求批量导出，DES 只设计了单条导出接口
- REQ 要求失败重试，DES 没有失败路径或异常策略
- REQ 依赖状态筛选，DES 中没有状态字段或状态流设计

失败处理：
- 会改变实现路径、接口设计、状态模型或主流程的，判 `Critical`
- 仍可继续写但返工概率高的，判 `Major`

### 4. 任务完整

检查 TASK 是否覆盖了 REQ 中全部关键功能点，以及支撑这些功能所需的必要实现动作。

重点检查：
- REQ 中每个关键功能点是否有任务承接
- 需要前后端联动的能力，TASK 是否覆盖完整链路
- 是否只有页面任务、没有接口任务，或只有接口任务、没有交互任务
- 是否只写了局部动作，没有覆盖主线路径

典型失败：
- “导出记录”在 REQ 中存在，但 TASK 中完全缺失
- 功能需要后端接口，TASK 只写前端页面调整
- 任务表只写展示，不写提交、校验或结果处理

失败处理：
- 主功能缺失判 `Critical`
- 支撑链路缺失判 `Major`
- 仅拆分粒度不理想但不影响主线开发，可判 `Warning`

### 5. 版本一致

检查 REQ / DES / TASK / CHK 的版本号、修订标记或同步状态是否一致。

重点检查：
- 文档版本号是否一致
- 最近一次关键变更是否同步回写到关联文档
- 是否存在 REQ 已更新、TASK / CHK / DES 仍停留在旧版的情况

典型失败：
- REQ v0.3，而 DES-前端仍是 v0.2
- TASK 已更新为新需求结构，但 CHK 还是旧验收口径

失败处理：
- 默认判 `Warning`
- 若版本不一致已经导致内容冲突、覆盖断层或错误门禁判断，升级为 `Major`

### 6. 文档齐套性

检查当前 scope 内是否具备最小完整 specs 集合。

重点检查：
- 模块是否同时存在 REQ / DES-前端 / DES-后端 / TASK / CHK
- 全局是否存在 `01-non-functional.md`
- 若项目进入多模块阶段，关键 NFR 是否已在相关模块 `CHK-*.md` 中得到承接
- 是否存在只有 TASK / CHK / DES 而无 REQ，或只有全局 NFR 定义但没有任何模块级验证落点的情况

典型失败：
- 模块已有 TASK / CHK，但缺少 REQ
- 模块已有前端 DES，但缺少后端 DES 且该模块明确需要后端支撑
- 项目已进入多模块阶段，但缺少全局 NFR 或没有任何模块 CHK 承接关键 NFR

失败处理：
- 阻断当前开发或破坏结构完整性的，判 `Critical`
- 明显削弱治理能力但不阻断当前局部开发的，判 `Major`

### 7. 模块覆盖地图

检查 requirement / design / task / checklist 目录中的模块集合是否一致。

重点检查：
- 是否所有模块都在四类目录中成套出现
- 是否存在只在某一类目录出现的模块
- 是否存在命名相近但路径不一致的模块集合

典型失败：
- `requirement/订单管理` 存在，但 `task/订单` 才有任务文档
- `design/用户管理` 存在，而 `checklist/用户管理` 缺失

失败处理：
- 造成模块链路无法追溯的，判 `Major`
- 仅是局部命名偏差但尚可理解的，可判 `Warning`

### 8. 跨模块依赖一致

检查 REQ 中声明的依赖模块是否存在、是否合理、是否形成可追溯关系。

重点检查：
- 依赖模块是否真实存在
- 依赖内容是否在被依赖模块中可定位
- 是否存在依赖已删除模块、未声明依赖或依赖关系明显失真

典型失败：
- REQ 声明依赖模块 A，但项目 specs 中不存在模块 A
- 被依赖模块未提供所声明接口/功能

失败处理：
- 会改变当前模块实现路径或集成路径的，判 `Critical`
- 会引发返工或集成误判的，判 `Major`

### 9. 全局 NFR / 模块验证覆盖

检查 `01-non-functional.md` 与相关模块 `CHK-*.md` 是否形成有效承接关系。

重点检查：
- 关键 NFR 是否在至少一个相关模块 `CHK` 中有对应检查项
- 模块 CHK 是否正确引用 NFR 编号或可追溯到相关 NFR
- 是否存在关键 NFR 完全无人承接

典型失败：
- NFR 中有安全要求，但所有相关模块 CHK 都无对应安全检查项
- 模块 CHK 引用了不存在的 NFR 编号，或完全没有 NFR 映射关系

失败处理：
- 关键全局质量门失效的，判 `Major`
- 仅覆盖粒度不足但主线仍可理解的，可判 `Warning`

### 10. 命名与路径规范

检查文件名、目录层级、编号格式是否符合 `01-spec-rules.md`。

重点检查：
- 文件名是否符合规范
- 模块路径是否一致
- 编号格式是否与规范冲突

典型失败：
- `REQ-用户.md` 与 `TASK-用户管理.md` 实际指向同一模块
- CHK 编号格式与规则定义不一致，导致追溯困难

失败处理：
- 影响追溯、自动定位或理解的，判 `Major`
- 仅局部格式不理想但不影响理解的，可判 `Warning`

## 子功能门禁判断维度（仅 `subfeature`）

完成一致性校验后，再判断当前模块或子功能是否允许进入 apply。

### 1. 主链路是否闭环

必须同时满足：
- REQ 中当前子功能的关键目标，DES 有支撑
- REQ 中当前子功能的关键目标，TASK 有承接
- REQ 中当前子功能的关键结果，CHK 可验证
- DES 中当前子功能的关键约束，TASK 有落实

### 2. 是否存在阻断开工的硬问题

以下任一存在，都不得直接判为“可开工”：
- 关键编号断链
- 主线验收缺失
- 关键设计契约不匹配
- 主功能任务未覆盖
- 会改变实现路径的关键未决项

### 3. 是否仍需开发者自行补关键规则

若仍要求开发者自行猜测以下内容，不得判为“可开工”：
- 功能边界
- 成功 / 失败路径
- 接口契约
- 状态流转
- 验收标准

### 4. 是否具备直接开发与直接验收的最低条件

必须满足：
- TASK 足够指导当前开发
- CHK 足够定义当前完成标准
- 当前实现不依赖大面积猜测

## 模块 / 全局健康判断维度（仅 `module` / `global`）

### 1. 结构是否齐套

必须判断：
- specs 基本文档集合是否完整
- 模块链路是否存在大面积缺口
- 是否存在孤儿文档或断层模块

### 2. 覆盖是否失衡

必须判断：
- requirement / design / task / checklist 是否明显失衡
- 全局 NFR 与验收清单是否失配
- 模块间依赖是否存在系统性未声明或错误声明

### 3. 同步是否可控

必须判断：
- 版本、状态、命名、路径是否仍在可维护范围内
- 是否存在会持续放大治理成本的结构性问题

### 4. 是否具备继续演进的最低条件

必须满足：
- 当前 specs 体系仍可定位、可追溯、可扩展
- 关键治理问题数量可控
- 不依赖大面积人工猜测来理解文档之间关系

## 必须确认问题规则

只有会改变以下内容的问题，才进入“必须确认”清单：

| 类型 | 是否进入必须确认 |
|---|---|
| 改变实现路径 | 是 |
| 改变范围边界 | 是 |
| 改变接口契约 | 是 |
| 改变验收口径 | 是 |
| 改变模块依赖路径 | 是 |
| 仅是文案优化 | 否 |
| 仅是编号格式不统一但不影响理解 | 否 |
| 仅是局部表达可改进 | 否 |

输出约束：
- 默认最多展开 2 个必须确认问题
- 若必须确认问题超过 2 个，说明当前 scope 仍未收敛，应倾向判为 `暂不可开工` 或 `不达标`

## 问题分级标准

| 级别 | 含义 | 是否阻断 apply / 是否影响健康 |
|---|---|---|
| Critical | 会直接改变当前范围、关键路径、接口契约、验收口径、依赖路径或主链路闭环 | 是 / 是 |
| Major | 不一定改变范围，但会导致明显返工、误开发、漏开发或治理失控 | 通常是 / 是 |
| Warning | 不影响当前主线开发，但说明文档存在同步问题、局部一致性问题或信息粒度偏弱 | 否 / 视情况 |
| Suggestion | 表达、结构、可维护性优化建议 | 否 / 否 |

说明：
- `Minor` 不再作为默认分级，统一用 `Warning` 表达“先不阻断，但值得注意”
- 若 Warning 已实质影响门禁判断或健康判断，应升级为 Major
- Suggestion 只在确有必要时保留，默认不展开

## 审查结论分级

### `subfeature`

| 结论 | 含义 | 条件 | 后续动作 |
|---|---|---|---|
| 可开工 | 当前 spec 已足以支撑直接开发 | 无 Critical；无阻断型 Major；无必须先确认问题 | 进入 `/zsspec-apply <模块>` |
| 有条件可开工 | 存在少量非阻断问题，但主线清晰 | 无 Critical；Major 不影响主线；必须确认问题为 0 或已即时澄清 | 先轻量修订，再进入 apply |
| 暂不可开工 | 存在阻断问题或核心未决项 | 任一 Critical；或阻断型 Major；或存在必须先确认问题 | 转 `zsspec-change`、`zsspec-spec` 或 `zsspec-brain` |

### `module` / `global`

| 结论 | 含义 | 条件 | 后续动作 |
|---|---|---|---|
| 健康 | 当前 specs 体系结构完整、链路清晰、治理可控 | 无 Critical；无系统性 Major；关键链路齐套 | 继续维护；必要时收敛到局部 gate verify |
| 有风险 | 存在非致命但值得尽快修复的治理问题 | 无 Critical；存在可控 Major 或多项 Warning | 优先修订 specs，再继续扩展 |
| 不达标 | 存在结构性缺口、关键断链或治理失控问题 | 任一 Critical；或系统性 Major；或关键文档缺失 | 转 `zsspec-change`、`zsspec-spec` 或 `zsspec-brain` |

## 默认读取顺序

### `subfeature`
1. 用户当前需求、本轮补充要求、最近明确确认的口头约束
2. REQ 的当前子功能 Requirement Core List
3. DES 的当前子功能核心设计要点
4. TASK 的当前子功能 Task Core List 与任务总表
5. CHK 的当前子功能 Check Core List 与证据要求
6. 必要详情区、补充约束、接口定义
7. 仅当 spec 明确依赖现有实现或用户要求时，再读取相关目录、组件、接口或既有实现

### `module`
1. 当前模块的 REQ / DES / TASK / CHK
2. 当前模块各 Core List 与状态区
3. 当前模块依赖关系声明、版本信息、变更记录
4. 仅在必要时补读详情区与相关局部说明

### `global`
1. `01-spec-rules.md`
2. `01-non-functional.md`
3. requirement / design / task / checklist 四类目录的模块集合
4. 各模块 REQ / DES / TASK / CHK 的最小必要头部信息与 Core List
5. 必要时抽查关键模块 CHK 中的非功能检查项
6. 仅在某个关键问题需要确认时，再补读具体详情

读取原则：
- 先读最小集合，后补读必要详情
- `global` 先看结构地图，再下钻模块
- 不得默认扫完整模块历史与全量实现
- 任何扩展读取都必须服务于当前判断

## 执行动作

### `subfeature`
1. 明确当前模块与本次检查的子功能范围
2. 读取用户当前诉求与 REQ / DES / TASK / CHK 的最小必要内容
3. 按五类基础一致性维度执行校验：编号追溯、需求覆盖、设计一致、任务完整、版本一致
4. 汇总失败项、警告项与必须确认项
5. 判断这些问题是否阻断当前模块或子功能进入 apply
6. 输出固定格式的一致性检查报告与下一步建议

### `module`
1. 明确当前模块整体范围
2. 检查当前模块文档齐套性、覆盖完整性、版本与状态同步情况
3. 汇总结构性问题、风险项与建议修订点
4. 输出模块级健康检查报告与下一步建议

### `global`
1. 先建立全局 specs 结构地图
2. 检查文档齐套性、模块覆盖地图、全局 NFR / 验收清单、命名路径规范、依赖一致与版本治理
3. 汇总结构性问题、系统性风险与建议修订路径
4. 输出全局健康检查报告与下一步建议

若命中以下情况，可轻量扩展检查：
- spec 明确写了“复用现有逻辑/组件/接口”
- spec 明确要求兼容旧行为、旧数据或旧接口
- 用户明确要求核对与现有实现是否冲突

但即使扩展检查，也必须保持“当前 scope 优先”，不得退化为全面代码审查。

## 输出要求

- 必须先给结论，再给问题
- 失败项逐条列出，便于用户快速定位
- 警告项默认折叠或尽量少写，不得喧宾夺主
- 必须确认问题默认最多 2 个
- 不得输出长篇分维度议论文式报告
- 不得为了“显得全面”而扩写非阻断问题
- 不得把 verify 写成 code review、架构优化报告或仓库巡检报告
- `subfeature` 无阻断项时，应明确写出“可开工”
- `module` / `global` 无结构性风险时，应明确写出“健康”

## 建议追问用户的典型问题

当 verify 发现关键未决项时，应优先整理成少量明确问题，而不是泛泛提醒。可参考以下类型：

| 类型 | 典型问题 |
|---|---|
| 范围边界 | 这次只做 [子功能]，是否明确不包含 [相邻子能力]？ |
| 成功 / 失败路径 | 失败时是回滚、保留中间态，还是允许部分成功？ |
| 权限规则 | 哪些角色可见、可改、可触发？ |
| 接口契约 | 字段是否允许为空？错误返回是否需要稳定约定？ |
| 验收口径 | 什么证据算通过？截图、日志、接口响应还是测试结果？ |
| 模块依赖 | 当前依赖是否应落到 [模块A]，还是应拆出新模块？ |

若这些问题会改变当前实现路径、依赖路径或验收口径，必须进入“必须确认”清单。

## 与其他阶段的衔接

- spec 生成完成后，可先执行模块级或全局级 verify 做 specs 健康检查
- apply 开始前，必须执行 `subfeature` 级 verify 或等价的局部门禁检查
- spec 有更新后，应重新执行 verify
- verify 结论为“可开工”或“有条件可开工”后，才允许进入 `/zsspec-apply <模块>`
- verify 发现编号、覆盖、版本等小范围问题 → `/zsspec-change <模块> [修订说明]`
- verify 发现设计链路缺口、文档齐套性问题或结构性覆盖断层 → `/zsspec-spec <模块> --update`
- verify 发现边界、方案、依赖或模块拆分未收敛 → `/zsspec-brain <模块>`

## 触发时机

| 时机 | 建议级别 | 推荐 scope |
|---|---|---|
| spec 生成完成后 | 建议立即触发 | `module` 或 `global` |
| apply 开始前 | 强制通过 | `subfeature` |
| 用户手动要求检查 | 随时可触发 | 按目标选择 |
| 多模块阶段性收口 | 强烈建议 | `global` |

## 关键原则

- verify 先做一致性校验，再做结论判断
- verify 的重点是阻断硬问题与治理关键问题，不是罗列所有可优化项
- verify 默认仍以当前准备开工的模块或子功能为优先
- verify 输出必须简洁、明确、可执行
- verify 只要发现当前实现仍依赖大面积猜测，就不得允许进入 apply
- 全局 verify 只做 specs 体系治理，不默认扩展为代码现实审计
- 非阻断问题默认后置，不逐条展开
- 仅当 spec 明确依赖现有实现或用户明确要求时，才扩展仓库现实 / 现有实现检查
