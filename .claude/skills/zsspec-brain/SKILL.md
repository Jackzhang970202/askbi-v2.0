# ZSSpec Brain - 需求讨论

`zsspec-brain` 用于把模糊需求收敛为可进入 `zsspec-spec` 的结构化讨论结论。

## 职责边界

`zsspec-brain` 只负责：
- 探索当前项目上下文，避免脱离仓库现状讨论
- 识别输入形式，提取背景、范围、约束、优先级、成功标准
- 判断是否需要拆分；若需求过大，先收敛模块入口与当前推进模块
- 若存在多种合理路径，给出 2-3 个方案并明确推荐
- 输出结构化讨论结论，等待用户明确指令后再进入 `zsspec-spec`

不负责：
- 直接生成 REQ/TASK 文档
- 编写代码或执行实现
- 处理变更或验收

## HARD GATES

<HARD-GATE>
在呈现设计并获得用户认可之前，禁止调用任何实现技能、编写任何代码或执行任何实现操作。
</HARD-GATE>

<HARD-GATE>
在需求讨论完成后，禁止自动调用 `zsspec-spec`。必须等待用户明确发出"可以生成 spec""开始生成 spec""进入 spec"等指令后，才允许进入 `zsspec-spec`。
</HARD-GATE>

## 使用方式

```bash
/zsspec-brain <需求描述>
/zsspec-brain <模块>
```

## 最小流程

1. 探索当前项目上下文，避免脱离仓库现状讨论
2. 识别输入形式，提取背景、范围、约束、优先级、成功标准
3. 判断是否需要拆分；若需求过大，先收敛模块入口与当前推进模块
4. 若存在多种合理路径，给出 2-3 个方案并明确推荐
5. 输出结构化讨论结论，等待用户明确指令后再进入 `zsspec-spec`

## 命名职责边界

brain 阶段负责收敛当前推进模块的语义边界，并给出推荐的模块路径命名：
- 推荐模块路径使用 `{一级模块}-{二级模块}-...-{N级模块}`
- 未拆分的小需求可使用单段模块名
- 已拆分需求应输出可复用的模块路径建议

brain 只负责收敛模块路径语义和模块间关系；最终文件命名、目录结构和编号格式由 `zsspec-spec` 按规范落实。

## 进入 spec 条件

进入 `zsspec-spec` 前，必须同时满足：
- 范围已聚焦到一个模块级 spec
- 背景、边界、成功标准、明确不做项已收敛
- 不再存在关键双义性或未取舍方案
- 模块间关系已明确

## 结构化讨论结论

brain 阶段产物为结构化讨论结论，必须包含：

### 未拆分时

```markdown
## 结构化讨论结论

### 1. 目标 / 背景
- 为什么要做这个需求：
- 当前业务问题：
- 希望达到的结果：

### 2. 核心问题
- 本次讨论主要解决的问题：
- 当前最大的约束或难点：

### 3. 推荐方案
- 推荐方案名称：
- 选择理由：
- 为什么不选其他方案：

### 4. 范围内内容
- 本次 spec 需要覆盖的内容：
  - [内容1]
  - [内容2]

### 5. 明确不做的内容
- 本次不纳入 spec 的内容：
  - [不做1]
  - [不做2]

### 6. 成功标准
- 需求完成后，至少应满足：
  - [标准1]
  - [标准2]

### 7. 待确认事项
- [ ] [待确认事项1]

### 8. 建议模块名
- 建议模块名：
- 命名理由：
```

## 关键原则

- 默认一次推进一个关键问题，避免无关发散
- 能直接收敛时不强制追问
- 复杂需求可分段呈现并逐步确认
- 边界不清、范围过大或方案未收敛时，不得进入 `zsspec-spec`

## 阶段输出格式

阶段收口时使用固定四段输出：

```markdown
【本阶段总结】
- 已完成需求讨论
- 输出结构化结论

【当前结论/产物】
- 结构化讨论结论（见上文）
- 是否可进入 spec：[是/否]

【风险或待确认事项】
- [待确认项]

【下一步建议】
- 进入 spec 生成：/zsspec-spec <模块>
- 继续讨论：[具体方向]
```
