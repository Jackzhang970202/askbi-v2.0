## REQ-bi-query-自然语言问数

**版本**: v1.1.0 | **状态**: 未开始 | **优先级**: P0

### 需求描述
用户通过自然语言提问，系统自动生成 SQL 查询业务数据库，并按统一回答结构返回问数结果，默认包含问题结果/结论、回答依据/佐证、数据图表；当用户勾选“分析解读”时，额外输出业务视角的分析解读。

### 前置条件
- 用户已登录
- 已配置 PostgreSQL/MySQL 数据源
- 已生成可用元数据 (schema_data)
- 已存在有效 chat_id

### 输入
- 自然语言问题
- 数据源名称
- 会话 ID (chat_id)
- 是否输出分析解读（可选开关，默认关闭）

### 输出
- `summary`: 主回答正文，按统一结构组织
- `sql`: 最终执行 SQL
- `tables`: 本次查询涉及的数据表名列表
- `chart`: ECharts option 或 `{"chart_needed": false, "reason": "..."}`
- `thoughts`: 过程性提示或轮次说明

### 输出结构要求
1. **问题结果、结论**
   - 必须直接回答用户问题
   - 对指标问法需明确给出数值、增量、增速、排名、最值、结论月份等关键信息
   - 该部分应位于回答最前面
2. **回答依据、佐证**
   - 必须包含支撑前述结论的详细数据或明细表
   - 可同时补充必要的推理说明，但不得替代结论部分
3. **数据图表**
   - 系统需根据结果数据自主选择合适图表类型
   - 若用户明确要求图表形式，优先按用户要求生成
   - 若数据不适合图表，可返回不生成图表的原因
4. **分析解读（可选）**
   - 仅当用户显式勾选时输出
   - 内容为业务视角的客观解读，不是重复结论，也不是结论推导过程
5. **建议模块**
   - 不强制单独成段
   - 如有必要，可融合进分析解读部分

### 处理规则
1. 加载数据源元数据（优先 DB，回退 refer/ 文件）
2. 构建 prompt，包含用户问题、数据源名称、表结构元数据
3. 调用 LLM 生成 PostgreSQL/MySQL 可执行查询 SQL
4. 校验 SQL 仅为 SELECT/WITH 查询，拦截非查询操作
5. 执行 SQL，获取结果数据
6. 调用 LLM 生成主回答 `summary`，并按“结果/结论 → 依据/佐证 → 图表说明 → 可选分析解读”的顺序组织
7. 调用 LLM 生成 `chart`
8. 保存会话消息与请求记录，历史消息中的 `structuredData` 需保留 `summary/sql/tables/chart/thoughts`

### 验收标准
- [ ] 自然语言问题能生成可执行 SELECT/WITH SQL
- [ ] 非查询 SQL 被拦截并返回错误
- [ ] `summary` 首段直接回答问题结果或结论
- [ ] `summary` 包含可支撑结论的详细数据/佐证部分
- [ ] 有合适数据时生成图表，无合适数据时返回 `chart_needed: false`
- [ ] 未勾选分析解读时不输出该部分
- [ ] 勾选分析解读时输出业务视角解读
- [ ] 会话消息与请求记录正确保存

## REQ-bi-query-会话管理

**版本**: v1.0.0 | **状态**: 已完成 | **优先级**: P0

### 需求描述
支持 BI 问数会话的创建、列表查看、消息查看与会话删除。

### 前置条件
- 用户已登录

### 输入
- 创建会话: knowledge_id, datasource_name
- 查看列表: 无
- 查看消息: chat_id
- 删除会话: chat_id

### 输出
- 创建: chat_id, knowledge_id
- 列表: 会话列表（过滤掉 `excel_` 开头的会话）
- 消息: 消息列表（含 `structuredData`）
- 删除: 成功/失败

### 处理规则
1. 创建会话时自动生成 chat_id（`bi_` 前缀）
2. 列表时过滤仅 BI 类型会话（chat_id 不以 `excel_` 开头）
3. 用户只能查看自己的会话（admin/manager 可查看全部）
4. 删除会话时同步清理进度缓存与数据库记录
5. 历史消息中的结构化数据字段需与提问接口返回保持一致

### 验收标准
- [ ] 创建会话返回唯一 chat_id
- [ ] 列表仅返回当前用户的 BI 会话
- [ ] 消息列表包含用户问题、AI 回复与结构化数据
- [ ] 删除会话后数据库与进度缓存已清理
- [ ] admin/manager 可查看所有用户会话

## REQ-bi-query-进度推送

**版本**: v1.0.0 | **状态**: 未开始 | **优先级**: P1

### 需求描述
BI 问数任务执行过程中，前端可实时获取进度文本。

### 前置条件
- 用户已发起 ask 请求

### 输入
- chat_id, offset（偏移量）

### 输出
- 增量进度文本列表、done 状态、next_offset

### 处理规则
1. ask 请求开始时调用 progress_service.init(chatid)
2. 执行过程中按阶段追加进度文本
3. 任务完成后调用 progress_service.done(chatid)
4. 前端通过 `/progress?chatid=&offset=` 轮询获取增量进度
5. 流式展示文本以 `structuredData.summary` 为准

### 验收标准
- [ ] 进度文本按阶段追加（加载结构→生成 SQL→执行 SQL→生成报告→生成图表）
- [ ] done 标记在任务完成后置为 true
- [ ] offset 机制支持增量获取
- [ ] 异常情况下 done 仍被正确设置

## REQ-bi-query-SQL安全校验

**版本**: v1.0.0 | **状态**: 已完成 | **优先级**: P0

### 需求描述
确保 LLM 生成的 SQL 仅包含查询操作，防止数据篡改。

### 前置条件
- LLM 已生成 SQL

### 输入
- SQL 字符串

### 输出
- 校验通过的 SQL 或抛出异常

### 处理规则
1. 去除 SQL 两端反引号
2. 检查 SQL 是否以 SELECT 或 WITH 开头（不区分大小写）
3. 检测 SQL 中是否包含 insert/update/delete/alter/drop/create/truncate 关键字
4. 非查询 SQL 抛出 ValueError

### 验收标准
- [ ] SELECT 查询正常通过
- [ ] WITH ... SELECT 查询正常通过
- [ ] INSERT/UPDATE/DELETE 等语句被拦截
- [ ] 大小写变体被正确识别
