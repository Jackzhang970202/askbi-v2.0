## REQ-multi-agent-orchestration-统一会话模型

**版本**: v2.2.0 | **状态**: 进行中 | **优先级**: P0

### 需求描述
系统提供统一会话模型。用户点击“新建对话”后直接进入普通对话，同一会话内可后续绑定数据源、Excel 数据或团队能力，无需重新创建独立会话。

### 前置条件
- 用户已登录
- 会话服务可创建与读取会话
- 前端已加载可用数据源与团队列表

### 输入
- 新建对话操作
- 会话上下文变更操作：绑定/切换/清除数据源、Excel、团队
- 当前会话消息内容

### 输出
- 统一 chat_id
- 当前会话上下文信息
- 基于上下文路由后的回复结果

### 处理规则
1. 新建对话默认创建普通会话，不预先绑定 BI、Excel 或 Team 类型。
2. 会话上下文至少支持 `general`、`bi`、`excel`、`team` 四种类型。
3. 同一 chat_id 在生命周期内允许切换上下文，切换后仅影响后续消息路由，不删除历史消息。
4. 清除上下文后，会话立即退回普通对话。
5. 会话历史展示以统一会话为主，不再要求按 BI/Excel 预分裂创建入口。

### 验收标准
- [ ] 点击“新建对话”后直接进入普通对话界面
- [ ] 同一 chat_id 内可从普通对话切换到数据源问数
- [ ] 同一 chat_id 内可切换为团队协作
- [ ] 清除上下文后可继续普通对话
- [ ] 切换上下文不丢失已有聊天记录

## REQ-multi-agent-orchestration-普通对话链路

**版本**: v2.2.0 | **状态**: 进行中 | **优先级**: P0

### 需求描述
系统提供普通对话默认链路。当会话未绑定数据源、Excel 或团队时，消息应由普通对话智能体响应，而不是进入问数流程。

### 前置条件
- 已存在统一会话
- 普通对话智能体已注册或可由后端直接调用
- 模型配置可从全局配置读取

### 输入
- chat_id
- 用户消息
- 可选技能列表

### 输出
- 普通对话回复
- 普通对话消息持久化结果

### 处理规则
1. 未绑定任何执行上下文时，发送消息必须走普通对话链路。
2. 普通对话链路不可依赖 schema、datasource_name 或 workflow 前置条件。
3. 普通对话同样支持消息持久化和历史恢复。
4. 普通对话支持启用技能注入，但不强依赖技能存在。
5. 普通对话失败时直接返回错误，不自动降级到 BI/Excel/Team。

### 验收标准
- [ ] 未绑定上下文时发送消息得到普通对话回复
- [ ] 普通对话消息写入会话历史
- [ ] 普通对话不触发 SQL/Excel/Team 执行链路
- [ ] 历史会话恢复后可继续普通对话

## REQ-multi-agent-orchestration-会话内上下文挂载

**版本**: v2.2.0 | **状态**: 进行中 | **优先级**: P0

### 需求描述
系统在聊天输入区域提供“加号”入口，允许用户在当前会话内挂载数据源、Excel 数据源或团队，驱动后续消息进入对应执行链路。

### 前置条件
- 当前存在有效 chat_id
- 数据源与团队列表可查询
- 前端聊天输入区域可扩展操作入口

### 输入
- 选择数据库数据源
- 选择 Excel 数据源
- 选择团队
- 清除当前上下文

### 输出
- 当前会话上下文展示信息
- 新的消息路由状态

### 处理规则
1. 选择数据库数据源后，会话上下文变为 `bi`，后续消息走 BI workflow。
2. 选择 Excel 数据源后，会话上下文变为 `excel`，后续消息走 Excel workflow。
3. 选择团队后，会话上下文变为 `team`，后续消息走 TeamCoordinator。
4. 同一时刻会话只允许一个主上下文生效。
5. 切换上下文时应在 UI 明确展示当前生效上下文。

### 验收标准
- [ ] 输入框存在加号入口并可打开上下文选择菜单
- [ ] 可在当前对话内绑定数据库数据源
- [ ] 可在当前对话内绑定 Excel 数据源
- [ ] 可在当前对话内绑定团队
- [ ] UI 正确显示当前上下文并允许清除

## REQ-multi-agent-orchestration-消息路由与历史恢复

**版本**: v2.2.0 | **状态**: 进行中 | **优先级**: P0

### 需求描述
系统按会话当前上下文动态路由消息，并在历史恢复时同步恢复该上下文，使同一会话在刷新、切换、重进后仍保持正确执行语义。

### 前置条件
- 会话模型已支持上下文字段
- 前后端均可读写上下文状态
- 各执行链路接口已存在或可扩展

### 输入
- 当前 chat_id
- 会话上下文状态
- 用户消息

### 输出
- 路由到 general / bi / excel / team 的执行结果
- 恢复后的上下文显示状态

### 处理规则
1. 前端发送消息时必须基于当前会话上下文，而不是基于页面 tab 决定路由。
2. 恢复历史会话时必须同时恢复上下文类型与其引用对象。
3. team 会话历史不得再混入 BI 历史语义中。
4. 普通对话、BI、Excel、Team 均应进入统一历史列表，但显示不同上下文标签。
5. SSE 订阅路径根据当前上下文类型动态选择。

### 验收标准
- [ ] 发送消息时不再依赖 activeTab 选择执行链路
- [ ] 历史恢复后显示正确的上下文标签
- [ ] 团队会话不会作为 BI 会话错误展示
- [ ] 不同上下文的会话都能在统一历史中正常打开
- [ ] SSE 连接路径与当前上下文匹配

## REQ-multi-agent-orchestration-图表分析体验

**版本**: v2.2.1 | **状态**: 未开始 | **优先级**: P1

### 需求描述
系统在统一聊天页中提供卡片化图表分析体验。图表展示、标题信息、快捷操作与配置面板需要形成稳定的信息层级，避免默认图表组件的工程化外观，提升 BI / Excel / Team 分析结果的可读性与产品感。

### 前置条件
- 聊天消息可返回结构化图表数据
- 前端存在图表渲染组件与消息卡片容器
- 用户可在结果卡片上执行查看、切换、配置等操作

### 输入
- 图表标题、图表类型、字段映射、图例与网格等显示配置
- 图表数据序列与分组信息
- 用户在图表卡片上的查看与配置操作

### 输出
- 卡片化图表结果视图
- 统一的图表主题与视觉层级
- 可编辑的图表配置面板

### 处理规则
1. 图表必须放入独立结果卡片中，不允许以裸图形式直接贴在消息正文区域。
2. 图表卡片至少拆分为图表区、底部信息条、配置面板三区域，图表区只承载核心可视化。
3. 标题、图例、操作按钮、配置表单必须遵循统一的间距、字号、圆角、边框和弱化/强调规则。
4. 默认配色必须来源于预设主题色板，不允许按序列随机生成颜色。
5. 坐标轴、网格线、图例、数据点等辅助元素必须弱化，主数据线/柱/面积为视觉焦点。
6. 图表详情视图支持表格 / 图表切换，并提供字段映射、风格、主题色、图表选项等配置入口。
7. 选中态只允许一个主强调信号，优先使用外层 ring 或等效高亮，不叠加多个竞争性强调样式。

### 验收标准
- [ ] 图表结果以卡片形式展示，存在清晰的图表区与底部信息条
- [ ] 标题与操作按钮不再挤占图表绘图区
- [ ] 主题色来源于固定色板，跨图表风格一致
- [ ] 图例、坐标轴、网格线的视觉权重低于主数据图形
- [ ] 图表详情存在表格/图表双视图与配置面板
- [ ] 选中图表时高亮明确但不过度喧宾夺主
