# AskBI 超大规模表元数据检索与 SQL 生成分级改进方案

**版本**: v1.0  
**日期**: 2026-06-15  
**适用模块**: BI 问数 / Excel 问数（以 BI 问数为主）

---

## 1. 背景与问题定义

当前 AskBI 的 SQL 生成链路依赖把数据源的元数据 JSON（表结构、字段、样例数据）直接或近似直接提供给大模型。

当数据源规模较大时，这种方式会出现以下问题：

| 问题 | 表现 | 后果 |
|---|---|---|
| 上下文超长 | 表过多、字段过多、样例数据过多 | 模型输入超长、截断、成本升高 |
| 召回噪声大 | 模型面对大量无关表 | 选错表、漏表、误联表 |
| 联表不稳定 | 模型自行猜测 join | 重复计数、错误聚合、性能差 |
| 样例污染 | 样例数据太多且不相关 | 模型被样例误导 |
| 推理不分级 | 小库和超大库用同一策略 | 简单场景浪费，复杂场景失效 |

因此，需要把“全量元数据喂模型”的策略改造成“本地检索 + 分层召回 + 联表规划 + 分级门槛”。

---

## 2. 总体目标

| 目标 | 说明 |
|---|---|
| 控制上下文长度 | 不再把全量元数据直接给模型 |
| 提高选表准确率 | 先由系统召回候选表，再让模型生成 SQL |
| 稳定联表质量 | 系统先规划 join path，模型只负责表达 SQL |
| 支持分级策略 | 100 / 1000 / 10000+ 表使用不同方案 |
| 兼顾性能与成本 | 小规模场景保持简单，大规模场景按需升级 |

---

## 3. 核心设计原则

| 原则 | 说明 |
|---|---|
| 系统先选，模型后写 | 由系统检索候选表/字段，模型不直接扫描全库 |
| 两阶段以上推理 | 先选表，再选字段，再生成 SQL |
| 关系图先行 | 联表路径由系统规划，不由模型自由猜测 |
| 样例按需裁剪 | 样例只给相关字段和少量行 |
| 按规模分级 | 不同表数量区间启用不同复杂度策略 |
| 结果可审计 | 每一步保留检索、筛选、规划、生成证据 |

---

## 4. 分级策略总览

### 4.1 表规模分级

| 等级 | 表数量 | 推荐策略 | 复杂度 |
|---|---:|---|---|
| L1 | ≤ 100 | 轻量裁剪 + 单轮生成 | 低 |
| L2 | 101 ~ 1000 | 表级召回 + 字段级裁剪 + 两阶段生成 | 中 |
| L3 | 1001 ~ 10000 | 混合召回 + Join Graph + 三阶段生成 | 高 |
| L4 | > 10000 | 域路由 + 分片召回 + Join 子图规划 + 受控 SQL 生成 | 很高 |

### 4.2 触发门槛建议

| 条件 | 建议动作 |
|---|---|
| 表数 ≤ 100 且字段总数可控 | 允许直接裁剪后进模型 |
| 表数 > 100 | 必须先做表级召回 |
| 表数 > 1000 | 必须启用 Join Graph |
| 表数 > 10000 | 必须先做领域路由，禁止全库级召回 |
| 问题涉及多实体、多指标、多维度 | 强制进入联表规划模式 |

---

## 5. 各等级详细策略

## 5.1 L1：≤ 100 表

### 适用场景

| 条件 | 说明 |
|---|---|
| 表较少 | 一次性遍历成本可接受 |
| 联表有限 | 多数问题只涉及 1~3 张表 |
| 元数据体积不大 | 裁剪后仍能稳定进入模型上下文 |

### 推荐链路

| 步骤 | 动作 |
|---|---|
| 1 | 读取全量元数据 JSON |
| 2 | 基于问题做轻量关键词过滤 |
| 3 | 保留 Top 10~20 张候选表 |
| 4 | 每张表只保留关键字段和少量样例 |
| 5 | 单轮生成 SQL |
| 6 | 执行 + 校验 + 修正 |

### L1 上下文构成建议

| 内容 | 控制方式 |
|---|---|
| 表名/注释 | 全保留 |
| 字段 | 每表保留 10~30 个关键字段 |
| 样例 | 每表最多 1~2 行 |
| Join 信息 | 简单主外键提示 |

### L1 风险

| 风险 | 应对 |
|---|---|
| 表虽然不多但字段极多 | 增加字段裁剪器 |
| 样例数据太长 | 按字段相关性截断 |

---

## 5.2 L2：101 ~ 1000 表

### 适用场景

| 条件 | 说明 |
|---|---|
| 表数中等 | 已不能让模型自己选表 |
| 业务域开始明显分化 | 如人口、场所、楼宇、事件 |
| 联表场景增加 | 需要更稳定召回 |

### 推荐链路

| 步骤 | 动作 |
|---|---|
| 1 | 全量元数据预处理生成“表摘要卡片” |
| 2 | 用户问题 → 意图解析（指标/维度/实体/时间） |
| 3 | 表级召回（关键词 + embedding） |
| 4 | 候选表字段裁剪 |
| 5 | 第二轮生成 SQL |
| 6 | 执行 + 验证 |

### 必须新增的资产

| 资产 | 内容 |
|---|---|
| 表摘要卡片 | 表名、注释、业务关键词、关键字段、时间字段、指标字段、维度字段 |
| 字段索引 | 字段名、注释、所属表、字段类别 |
| 业务词典 | 街道、社区、人员、场所、GMV、口径等业务词映射 |

### L2 推荐模型提示输入

| 模块 | 内容 |
|---|---|
| 问题结构化意图 | 指标、时间、维度、过滤条件 |
| Top-K 候选表 | 3~10 张 |
| 候选字段 | 每表 5~15 个 |
| 裁剪样例 | 每表 1~2 行相关字段 |

### L2 风险

| 风险 | 应对 |
|---|---|
| embedding 召回不稳定 | 必须叠加关键词召回 |
| 事实表与维表混淆 | 引入表角色标签（事实/维/桥/汇总） |

---

## 5.3 L3：1001 ~ 10000 表

### 适用场景

| 条件 | 说明 |
|---|---|
| 表数很多 | 必须建立结构化检索与关系图 |
| 联表问题频繁 | 模型不能再自由猜 join |
| 相似表众多 | 如大量宽表、分区表、历史表 |

### 推荐链路

| 步骤 | 动作 |
|---|---|
| 1 | 元数据预处理为表图 + 字段图 + join graph |
| 2 | 问题先做领域路由 |
| 3 | 域内执行混合召回（关键词/BM25 + embedding + 业务词典） |
| 4 | 选事实表 |
| 5 | 自动补维表与桥表 |
| 6 | 规划最短 join path |
| 7 | 字段裁剪 |
| 8 | 生成 SQL |
| 9 | 执行 + 粒度校验 |

### 必须新增的结构

| 结构 | 用途 |
|---|---|
| Join Graph | 存主键、外键、推断关系、历史共现关系 |
| Table Role 标签 | 标识事实表/维表/桥表/汇总表 |
| Domain Map | 表到业务域映射 |
| SQL 经验库 | 历史问题 -> 成功 SQL -> 命中表 |

### L3 必须做的联表保护

| 保护项 | 说明 |
|---|---|
| 主统计对象识别 | 如人、场所、订单、记录 |
| 去重策略建议 | 自动建议 `COUNT(DISTINCT ...)` |
| 粒度冲突检测 | 防止 1:N join 后重复计数 |
| 维表补全 | 按过滤条件和展示维度补表 |

### L3 上下文不再包含

| 禁止项 | 原因 |
|---|---|
| 全量 schema JSON | 必超长 |
| 无关样例数据 | 纯噪声 |
| 模糊 join 候选全集 | 模型无法稳定选择 |

---

## 5.4 L4：> 10000 表

### 适用场景

| 条件 | 说明 |
|---|---|
| 超大库 | 企业级海量表、分层仓库、分主题域 |
| 大量同构表 | 分区表、快照表、历史表、ODS/DWD/DWS |
| 联表链路复杂 | 多层事实/维度/桥表组合 |

### 推荐链路

| 步骤 | 动作 |
|---|---|
| 1 | 先做领域分类路由 |
| 2 | 在命中领域内再做混合召回 |
| 3 | 候选事实表排序 |
| 4 | 生成最小 Join 子图 |
| 5 | 对字段进行二次裁剪 |
| 6 | 构造受控 SQL 提示包 |
| 7 | 生成 SQL |
| 8 | 执行校验，必要时自动回退修正 |

### L4 新增能力要求

| 能力 | 说明 |
|---|---|
| 域级白名单 | 先缩到 1~3 个业务域 |
| 表版本识别 | 过滤历史表/废弃表/快照表 |
| 语义聚类 | 同义表归并 |
| 强规则优先 | 优先使用人工维护的业务词典和 join 规则 |
| 多阶段 planner | 模型先输出 plan，再输出 SQL |

### L4 必须禁止

| 禁止项 | 原因 |
|---|---|
| 全库级 embedding 粗暴召回后直接进模型 | 噪声过高 |
| 让模型自行猜联表路径 | 错误率极高 |
| 直接使用全字段样例 | token 与噪声都不可控 |

## 6. 问题意图解析设计

### 6.1 解析目标

| 维度 | 示例 |
|---|---|
| 指标 | 数量、金额、占比、人数、总数、均值 |
| 维度 | 街道、社区、类别、时间、部门 |
| 实体 | 九小场所、人员、楼宇、企业 |
| 时间 | 今年、近三个月、某月、累计 |
| 过滤 | 丰台街道、状态=活跃 |
| 输出形态 | 排名、占比、趋势、明细 |

### 6.2 输出结构建议

```json
{
  "metrics": ["从业人员总数"],
  "dimensions": ["街道"],
  "entities": ["九小场所"],
  "filters": [{"field_hint": "street", "value": "丰台街道"}],
  "time_range": null,
  "output_type": "aggregate"
}
```

## 7. 元数据预处理方案

### 7.1 从全量 JSON 生成表摘要卡片

每张表生成一条摘要，不再把原始 JSON 直接给模型。

| 字段 | 说明 |
|---|---|
| table_name | 表名 |
| comment | 表注释 |
| keywords | 从表名/注释/字段提取的关键词 |
| time_columns | 时间字段 |
| metric_columns | 指标字段 |
| dimension_columns | 维度字段 |
| pk_columns | 主键字段 |
| fk_columns | 外键字段 |
| table_role | fact/dim/bridge/summary |
| domain | 业务域 |

### 7.2 字段级索引

| 内容 | 用途 |
|---|---|
| 字段名 | 精确匹配 |
| 字段注释 | 语义匹配 |
| 所属表 | 回溯定位 |
| 字段类别 | 指标/时间/维度/ID |
| 示例值摘要 | 辅助过滤判断 |

## 8. 联表查询规划方案

### 8.1 Join Graph 结构

| 字段 | 说明 |
|---|---|
| left_table | 左表 |
| left_column | 左字段 |
| right_table | 右表 |
| right_column | 右字段 |
| relation_type | fk / inferred / manual / history |
| confidence | 关系置信度 |
| cardinality | 1:1 / 1:n / n:n |

### 8.2 联表规划策略

| 步骤 | 动作 |
|---|---|
| 1 | 先选主事实表 |
| 2 | 根据过滤维度补维表 |
| 3 | 根据业务词典补桥表 |
| 4 | 使用最短路径或最小子图求解 join path |
| 5 | 生成 join 风险提示 |

### 8.3 粒度保护

| 风险 | 防护 |
|---|---|
| 1:N join 导致重复计数 | 识别主统计对象并建议 `COUNT(DISTINCT)` |
| 汇总表与明细表混用 | 表角色强约束 |
| 时间粒度冲突 | 时间字段粒度标记 |
| 维表重复映射 | join cardinality 校验 |

## 9. 模型上下文构建方案

### 9.1 最终给模型的上下文应包含

| 内容 | 数量控制 |
|---|---|
| 用户问题 | 1 条 |
| 意图结构化结果 | 1 份 |
| 候选事实表 | 1~3 张 |
| 候选维表 | 2~5 张 |
| join path | 1 套 |
| 候选字段 | 每表 5~15 个 |
| 样例数据 | 相关字段 1~2 行 |
| 风险提示 | 1 组 |

### 9.2 不应再直接给模型的内容

| 内容 | 原因 |
|---|---|
| 全量 schema JSON | 超长 |
| 全字段样例 | 噪声大 |
| 全库表清单 | 模型无法稳定选择 |

## 10. 推荐检索打分模型

### 10.1 混合打分

| 信号 | 权重建议 | 说明 |
|---|---:|---|
| 业务词典命中 | 0.30 | 最稳定 |
| BM25 / 关键词 | 0.25 | 精确召回 |
| embedding 相似度 | 0.20 | 语义补充 |
| 历史 SQL 共现 | 0.15 | 越用越准 |
| Join 图邻接 | 0.10 | 自动补表 |

### 10.2 输出

| 输出 | 说明 |
|---|---|
| Top-K 候选表 | 进入字段裁剪 |
| 表角色 | 事实/维/桥 |
| 推荐 join path | 给 SQL 生成阶段 |
| 风险提示 | 给校验器 |

## 11. 系统模块改造建议

### 11.1 新增模块

| 模块 | 作用 |
|---|---|
| MetadataIndexer | 从全量 JSON 构建摘要卡片与字段索引 |
| DomainRouter | 领域路由 |
| TableRetriever | 表级混合召回 |
| JoinPlanner | 联表路径规划 |
| FieldReducer | 字段裁剪与样例裁剪 |
| QueryPlanner | 汇总上述结果，形成 SQL prompt 包 |
| ResultSanityChecker | 执行后结果合理性校验 |

### 11.2 与现有 AskBI 代码结合点

| 位置 | 建议改造 |
|---|---|
| `core/schema_loader.py` | 增加元数据摘要输出接口 |
| `backend/ask/workflows/bi_workflow.py` | 在 SQL 生成前加入检索规划阶段 |
| `refer/*_metadata.json` 使用链路 | 从“直接给模型”改为“先索引、后裁剪” |
| `utils` 或 `services` | 新增索引器、召回器、JoinPlanner |

## 12. 分阶段实施路线

### 阶段 1：低风险改造（优先）

| 目标 | 内容 |
|---|---|
| 替换全量 schema 输入 | 先做表摘要卡片 |
| 控制样例长度 | 只保留相关字段和 1~2 行样例 |
| 表级召回 | 关键词 + 业务词典 Top-K |

### 阶段 2：中等规模增强

| 目标 | 内容 |
|---|---|
| embedding 召回 | 引入表摘要向量索引 |
| 字段裁剪 | 二阶段生成 |
| 历史 SQL 经验库 | 提高召回命中率 |

### 阶段 3：大规模联表支撑

| 目标 | 内容 |
|---|---|
| Join Graph | 主外键 + 推断关系 + 人工关系 |
| 主事实表识别 | 粒度约束 |
| 最小 Join 子图 | 减少模型猜测 |

### 阶段 4：超大规模治理

| 目标 | 内容 |
|---|---|
| 领域路由 | 先按业务域分桶 |
| 表版本过滤 | 剔除历史表/快照表 |
| Planner 模式 | 先 plan 后 SQL |

---

## 13. 风险与注意事项

| 风险 | 说明 | 处理 |
|---|---|---|
| 人工词典维护成本 | 业务词映射需要持续维护 | 先覆盖高频词 |
| Join 图不完整 | 外键不一定完备 | 支持人工补关系 |
| 事实表识别错误 | 会导致 SQL 基表错误 | 增加历史成功 SQL 反馈 |
| 领域路由误判 | 会漏召回候选表 | 允许跨域补召回 |
| 上下文过度裁剪 | 可能漏字段 | 加回退机制 |

---

## 14. 最终建议

| 场景 | 建议 |
|---|---|
| 当前表数不大但增长很快 | 立即上 L2 方案 |
| 已有联表错误、重复计数问题 | 立即补 Join Graph |
| 预计会到 1000+ 表 | 不要继续优化 prompt，直接做检索式架构 |
| 预计到 10000+ 表 | 必须加领域路由和 join 子图规划 |

---

## 15. 一句话结论

当表达到 100 / 1000 / 10000+ 规模时，SQL 生成系统必须从“全量 schema prompt”升级为“分级检索 + 关系图规划 + 裁剪上下文 + 受控生成”的架构；系统负责选表和联表，模型负责表达 SQL。
