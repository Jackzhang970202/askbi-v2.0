# ZSSpec Change - 变更处理

用于判断当前变更应在当前阶段修订，还是回退到 `zsspec-brain` 或 `zsspec-spec` 重新处理。

## 职责边界

`zsspec-change` 只负责：
- 分析变更影响范围
- 按影响程度分类变更
- 决定处理方式：直接修正、修订后继续、或回退到更早阶段
- 执行相应的文档和代码修正

不负责：
- 生成全新的需求讨论（由 `zsspec-brain` 负责）
- 从头生成 spec（由 `zsspec-spec` 负责）
- 验收判定（由 `zsspec-done` 负责）

## HARD GATES

<HARD-GATE>
在完成变更分类前，禁止直接进入代码实现。
</HARD-GATE>

<HARD-GATE>
影响 REQ / TASK 的变更，优先先修文档，再继续开发或验收。
</HARD-GATE>

<HARD-GATE>
影响模块边界的变更，必须回退到 `zsspec-brain` 重新收敛。
</HARD-GATE>

## 使用方式

```bash
/zsspec-change <模块> [变更描述]
```

## 变更分类

| 类型 | 判断标准 | 处理方式 | 去向 |
|------|----------|----------|------|
| **文档纠错** | 改后不新增功能、不改变范围、不改变设计、不新增任务 | 直接修正文档 | 当前阶段继续 |
| **spec 补充** | 属于当前模块原边界内，漏了小需求或验收项 | 修订当前模块 spec | 继续当前阶段 |
| **小范围变更** | 改局部设计、流程或任务结构，仍属于当前模块边界 | 修订 spec，重开受影响任务 | 重新 apply |
| **重大变更** | 原理解有误、方案需重选、边界要调整 | 回退到 `zsspec-brain` | 重新 brain |
| **Bug 修复** | 实现与 spec 不符，或 spec 本身错误 | 判断影响：小影响直接修，大影响回退 | 视情况定 |

## 版本同步规则

所有涉及文档内容变更的变更类型（文档纠错除外），必须在修正后同步更新版本号：

| 变更类型 | 版本操作 | 影响文档 |
|----------|----------|----------|
| 文档纠错 | 不更新版本号 | 仅修改错误内容 |
| spec 补充 | 次版本号 +1（v0.1 → v0.2） | 修订的 REQ / DES / TASK / CHK |
| 小范围变更 | 次版本号 +1 | 修订的 spec + 下游关联文档 |
| 重大变更 | 主版本号 +1（v0.x → v1.0） | 全部关联文档统一升级到新版本 |
| Bug 修复（小影响） | 次版本号 +1 | 仅修复涉及的文档 |
| Bug 修复（大影响） | 回退后按 brain/spec 规则生成新版本 | 按新文档处理 |

执行规则：

1. **先读源版本**：在处理变更前，先读取被修改文档的当前版本号
2. **按分类决定增量**：根据变更分类确定版本号增量（+0 / +0.1 / +1.0）
3. **同步下游文档**：若修改了 REQ，需检查 DES / TASK / CHK 是否需要同步更新版本号以保持一致
   - REQ 版本变更后，DES / TASK / CHK 若内容未变但需标注"已对齐 REQ v{x.y}"，在其变更记录中追加同步说明
   - 若 DES / TASK / CHK 内容也因本次变更需要修改，则同步升级版本号
4. **追加变更记录**：在文档末尾变更记录区写入本次变更的版本、日期、类型与摘要
5. **版本一致性校验**：变更完成后，确认同一模块的 REQ / DES / TASK / CHK 版本号处于一致或已声明兼容的状态

## 各类变更详细处理

### 1. 文档纠错

**判断标准**：
- 错别字、格式问题
- 不影响功能理解的描述修正
- 编号、路径等元数据修正

**处理**：
- 直接修正文档
- 不回退，不重新生成
- 当前阶段继续

### 2. spec 补充

**判断标准**：
- 属于原边界内的遗漏
- 不改变已有功能定义
- 不新增模块间依赖

**处理**：
- 修订 REQ / TASK
- 补充遗漏的功能点或检查项
- 继续当前阶段

### 3. 小范围变更

**判断标准**：
- 局部设计调整
- 流程微调
- 任务结构变化
- 仍在当前模块边界内

**处理**：
- 修订 spec
- 重开受影响任务
- 重新进入 apply 阶段

### 4. 重大变更

**判断标准**：
- 原理解有误
- 方案需重选
- 模块边界要调整
- 依赖关系变化

**处理**：
- 回退到 `zsspec-brain <模块>`
- 重新收敛需求
- 重新生成 spec

### 5. Bug 修复

**判断标准**：
- 实现与 spec 不符
- 或 spec 本身错误导致的问题

**处理**：
1. 分析 Bug 影响范围
2. 判断：
   - **小影响**（纯代码实现问题）：直接修复代码，轻量验证，更新检查项状态
   - **大影响**（涉及边界/功能定义）：回退到 brain 或 spec
3. 输出修复报告

## 在各阶段的处理

### brain 阶段执行 change

- 讨论方向调整 → 更新 brain 结论，继续 brain
- 范围重拆 → 重新 brain

### spec 阶段执行 change

- 文档纠错 → 直接修正，继续 spec
- spec 补充 → 修订后升级版本号，继续
- 小范围变更 → 修订 spec 并升级版本号，完成后继续或重新 apply
- 重大变更 → 回退到 brain

### apply 阶段执行 change

- 文档纠错 → 修正后继续开发
- spec 补充 → 停开发，修订 spec 并升级版本号，继续开发
- 小范围变更 → 停开发，修订 spec 并升级版本号，重新 apply
- 重大变更/Bug → 停开发，回退到 brain 或 spec

### review 阶段执行 change

- 纯实现缺陷 → 修复代码后重新 review
- spec 漏项 / 错项 → 修订 spec 后重新 review
- 边界 / 依赖 / 方案错误 → 回退到 brain 或 spec

### done 阶段执行 change

- TASK 未完成 / 未回填 → 回退到 `zsspec-apply`
- `automation=static` 未完成 / 未回填 → 回退到 `zsspec-code-review`
- `automation=auto` 未完成 / 未回填 → 回退到 `zsspec-e2e-run`
- `automation=manual` 未完成 → 保持在 done / 人工确认
- 文档/检查项定义问题 → 修正后重新 done，必要时回退到 `zsspec-test-gen` / `zsspec-spec`
- 实现缺陷 → 回退到 apply 修复
- 需求理解错误 → 回退到 brain

## 执行动作

1. 默认先读当前模块 REQ / TASK（含 Task Core List、设计概要、Check Core List）
2. 识别变更点、受影响范围
3. 按分类规则判断处理层级
4. 执行相应处理：直接修正、修订后继续、或回退
5. 若问题来自 review，明确是纯代码问题、spec 问题还是边界问题
6. 输出变更处理报告

## 阶段输出格式

阶段收口时使用固定四段输出：

```markdown
【本阶段总结】
- 变更类型：[类型]
- 影响范围：[范围]

【当前结论/产物】
- 处理方式：[处理方式]
- 已修正：[修正内容]

【风险或待确认事项】
- [待确认项]

【下一步建议】
- 继续当前阶段：[下一步]
- 回退到 brain：/zsspec-brain <模块>
- 重新 spec：/zsspec-spec <模块>
- 重新 apply：/zsspec-apply <模块>
```

## 关键原则

- 用户只描述变更，不需要自行判断回退层级
- 必须先分类，再决定处理方式
- 必须先修文档，再做代码或验收动作
- 边界变化不能通过局部打补丁绕过 brain
- 影响验收的变更必须补验证、补证据
- done 阶段若前序状态未收口，优先明确打回责任阶段，而不是在 done 中代替前序阶段补状态
