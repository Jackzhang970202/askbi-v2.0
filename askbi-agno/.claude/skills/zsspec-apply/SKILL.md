# ZSSpec Apply - 开始开发

确认当前模块规格文档已齐备且可执行，完成开发准入检查、状态更新，并在用户语义明确为开始实现时继续进入开发执行。

## 职责边界

`zsspec-apply` 只负责：
- 检查当前模块所需 REQ / TASK 是否齐全
- 完成开发准入检查
- 以 TASK 作为最小执行单位锁定当前开发任务
- 更新 REQ/TASK 状态为 `进行中`
- 启动编码实现（用户明确允许时）
- 在任务真实完成后立即回写对应 TASK 状态
- 在本轮 apply 收口前核对 TASK 状态是否与真实开发进度一致

不负责：
- 生成或修改 spec 文档
- 处理变更（由 `zsspec-change` 负责）
- 验收判定（由 `zsspec-done` 负责）

## HARD GATES

<HARD-GATE>
若仍存在关键歧义、文档冲突，或用户尚未明确允许进入实现，则不得直接开始开发。
</HARD-GATE>

<HARD-GATE>
apply 阶段的开发最小执行单位是 TASK。未锁定当前正在执行的 task，不得直接开始开发或扩展实现范围。
</HARD-GATE>

<HARD-GATE>
开发过程中每完成一个任务，应立即将对应 TASK 状态回写为 `已完成`；不得把常规状态回写累计到 `zsspec-done` 再统一补写。
</HARD-GATE>

<HARD-GATE>
本轮 apply 收口前，必须核对已完成任务是否都已在 TASK 文档中同步为 `已完成`；若存在漏同步，不得宣称当前开发阶段已收口。
</HARD-GATE>

## 使用方式

```bash
/zsspec-apply <模块>
```

## 进入条件

进入 apply 前，必须同时满足：
- REQ / TASK 文档齐全（TASK 含设计概要与检查清单）
- 已完成局部 gate verify（`scope=subfeature` 或等价局部门禁检查），且 verify 结论允许进入开发
- 用户已明确允许进入 apply；若要直接启动实现，还需具备明确的开始开发语义
- spec 生成后的执行计划概述已被用户看过
- 当前不存在需要回退 spec 阶段澄清的关键歧义
- 本轮实际要执行的 task 已能收敛到一个或多个明确任务项；若不能收敛，不得直接进入开发

若不满足任一条件：
- 不得进入开发阶段
- 必须先补齐文档或回到 spec 阶段澄清

## 执行动作

1. 检查当前模块所需 REQ / TASK 是否齐全
2. 按最小口径读取依据文档：REQ 读当前实现范围，TASK 读任务总表与当前任务，设计概要按任务范围读取，Check Core List 只读结构化检查项定义
3. 先锁定本轮实际执行的 task；若一次实现涉及多个 task，必须显式列出本轮任务集合
4. apply 启动时，将 REQ 中功能状态从 `未开始` 更新为 `进行中`
5. 仅将"实际开始执行"的 TASK 更新为 `进行中`
6. 若用户语义明确为开始实现，则仅围绕已锁定 task 进入编码实现；若实现范围超出 task，应先转 `zsspec-change`
7. 每个 task 达到完成条件后，立即将对应 TASK 状态回写为 `已完成`
8. 本轮开发告一段落时，执行 TASK 状态一致性核对，确认已完成/进行中/未开始状态与真实开发进度一致

## 状态回写规则

- 每完成一个任务，应立即将对应 TASK 状态回写为 `已完成`
- task 的"完成"以该任务对应实现已完成、已达到任务自身 DoD、且不存在阻断当前任务关闭的 blocker 为准
- 若一次实现涉及多个 task，必须逐个完成、逐个回写，不得等全部做完后统一补写
- 未完成的 task 只能保持 `进行中` 或 `未开始`，不得提前标记为 `已完成`
- 开发过程中的状态变化应即时回写，不应累计到 `zsspec-done` 再统一回写
- CHK 状态由对应验证阶段维护：`automation=static` 由 `zsspec-code-review` 回写，`automation=auto` 由 `zsspec-e2e-run` 回写，`automation=manual` 由 `zsspec-done` 或人工确认维护；apply 不提前修改 CHK 状态

## 进入 done 前的最低门禁

完成实现后、进入验收前，至少应满足：
- 已完成与本次变更对应的实现
- 已完成 `zsspec-code-review` 或等价的结构化代码审查
- 已完成必要的静态代码审查、安全审查、运行测试审查
- 不存在 blocker
- 已具备与本次变更相匹配的命令、日志摘要、截图或接口结果等验证证据

若任一条件不满足：
- 不得进入 `zsspec-done`
- 必须继续修复、补证据或重新审查

`zsspec-done` 只负责最终核验、补齐漏同步并判定 REQ 是否可更新为 `已完成`，不替代 apply 阶段的开发过程状态更新。

## 阶段输出格式

阶段收口时使用固定四段输出：

```markdown
【本阶段总结】
- 已完成开发准入检查
- 状态已更新为进行中
- 实际开发任务：[任务列表]

【当前结论/产物】
- 正在执行：[当前任务]
- 已完成：[已完成的任务]

【TASK状态核对】
- 已回写为已完成：[任务列表]
- 仍为进行中：[任务列表]
- 仍为未开始：[任务列表]
- 漏同步项：[无/列表]

【风险或待确认事项】
- [blocker]

【下一步建议】
- 继续开发：[下一步任务]
- 代码审查：/zsspec-code-review <模块>
- 完成验收：/zsspec-done <模块>
- 需要变更：/zsspec-change <模块> [变更描述]
```

## 阶段衔接

- 开发中需变更 → `/zsspec-change <模块> [变更描述]`
- 重要任务完成或准备收口 → `/zsspec-code-review <模块>`
- review 通过后进入验收 → `/zsspec-done <模块>`
