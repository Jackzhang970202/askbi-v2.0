## REQ-memory-management-用户画像记忆

**版本**: v1.0.0 | **状态**: 进行中 | **优先级**: P0

### 需求描述
系统需要基于用户历史对话自动总结长期用户画像记忆，用于后续问数、Excel 分析、普通对话和团队协作时提供个性化上下文。

### 前置条件
- 用户已登录，系统可获取 `user_id`
- 已完成 PostgreSQL 应用库表结构初始化
- LLM 配置可用于记忆抽取与摘要

### 输入
- 当前轮用户问题、助手回复、结构化结果、数据源、技能选择、上下文选项
- 该用户已有画像记忆

### 输出
- 用户级画像记忆记录
- 问答执行前可注入的画像上下文文本

### 处理规则
1. 用户级画像记忆按 `preference/background/constraint/goal` 分类。
2. 仅保存对跨会话有长期价值的信息，例如用户偏好、业务背景、常用口径、长期目标。
3. 每轮问答完成后异步抽取候选记忆，避免阻塞主问答链路。
4. 通过 `dedupe_key` 或语义相似度避免重复写入。
5. 画像记忆支持启用、归档和人工删除，不做物理删除作为默认策略。
6. 问答执行前读取当前用户活跃画像，拼接到系统上下文中。
7. PostgreSQL 表为权威存储；mem0 仅作为可选语义检索/向量检索层同步 `profile_text`。

### 验收标准
- [ ] 每轮问答后可生成或更新用户画像记忆
- [ ] 用户画像不会跨用户泄露
- [ ] 重复事实不会重复写入多条活跃记忆
- [ ] 后续对话可读取用户画像并注入模型上下文
- [ ] 记忆抽取失败不影响原始问答结果返回

## REQ-memory-management-会话记忆

**版本**: v1.0.0 | **状态**: 进行中 | **优先级**: P0

### 需求描述
系统需要为单次会话自动维护会话级记忆，用于记录该会话内的目标、主题、决策、状态和已确认口径，支持多轮问数连续性。

### 前置条件
- 会话已存在于 `askbi_chat_session`
- 用户已登录或会话已有归属用户

### 输入
- 当前会话全部或最近 N 轮消息
- 会话上下文：数据源、团队、技能、分析开关、记忆开关
- 当前轮结构化结果

### 输出
- 会话级记忆记录
- 后续同一会话可注入的会话上下文文本

### 处理规则
1. 会话记忆按 `goal/subject/decision/state` 分类。
2. 会话记忆只在当前 `chat_id` 内生效，不跨会话注入。
3. 会话删除时会话记忆同步归档或删除。
4. 切换数据源、团队、技能选择等上下文变化时写入 `state` 类型会话记忆。
5. 问答执行前优先注入当前会话记忆，再注入用户画像记忆。
6. 会话记忆允许自动过期，默认不影响用户画像。

### 验收标准
- [ ] 同一会话内后续问题能读取前序目标/口径/状态
- [ ] 不同会话之间不会共享会话记忆
- [ ] 删除会话时关联会话记忆被同步清理或归档
- [ ] 数据源/技能/上下文变化能被记录为会话状态

## REQ-memory-management-mem0集成

**版本**: v1.0.0 | **状态**: 未开始 | **优先级**: P1

### 需求描述
系统必须实际接入框架上层目录已有的 mem0 能力，记忆读写主链路必须通过 mem0 完成；保留现有向量存储方案，只将 mem0 当前基于 SQLite 的持久化改为 PostgreSQL。PostgreSQL 同时承载业务映射、审计和结构化补充存储。  

### 前置条件
- 已确认 `components/memory.py` 中 mem0 初始化方式
- 已配置或可配置 mem0 provider/mode

### 输入
- PostgreSQL 画像记忆与会话记忆的 `profile_text`
- mem0 配置项

### 输出
- mem0 中可检索的同步记忆
- 本地 PostgreSQL 记忆 ID 与 mem0 ID 的映射

### 处理规则
1. mem0 是记忆读写主链路，用户画像和会话记忆的新增、检索、更新必须实际调用 mem0。
2. 保留 mem0 现有向量存储方案，不在本次变更中替换向量库。
3. 如启用 mem0 OSS，必须替换其历史 SQLite 持久化方案，使底层元数据/历史存储改为 PostgreSQL。
4. PostgreSQL 同时保存业务映射、可编辑字段、审计事件和权限过滤辅助信息，但不得替代 mem0 主链路。
5. mem0 初始化失败、读写失败或不可用时，记忆相关能力必须显式报错，不允许静默降级为 PostgreSQL-only。
6. mem0 检索结果需与 PostgreSQL 权限过滤结果交叉校验，防止跨用户/跨会话泄露。

### 验收标准
- [ ] 用户画像和会话记忆的新增、检索、更新实际经过 mem0
- [ ] mem0 不可用时记忆相关能力明确报错，不发生 PostgreSQL-only 静默降级
- [ ] mem0 检索不会越权返回其他用户/会话记忆
- [ ] PostgreSQL 中保留 mem0 映射、审计和可编辑补充字段

## REQ-memory-management-管理与可视化

**版本**: v1.0.0 | **状态**: 未开始 | **优先级**: P1

### 需求描述
系统需要提供记忆管理页面，允许用户查看、搜索、启用/归档/删除自己的用户画像记忆和会话记忆，管理员可按权限查看全局记忆状态。

### 前置条件
- 用户已登录
- 记忆表和 API 已实现

### 输入
- 查询条件：记忆类型、范围、状态、关键词、会话 ID
- 管理操作：启用、归档、删除

### 输出
- 记忆列表、记忆详情、操作结果

### 处理规则
1. 普通用户只能管理自己的用户画像记忆和会话记忆。
2. admin/manager 可按用户筛选查看记忆，但不可绕过审计记录。
3. 删除默认执行逻辑删除/归档，保留事件审计。
4. 前端展示 `summary/profile_text/memory_kind/scope/status/update_time`。
5. 全局记忆模块列表展示用户画像记忆，支持手动修改 `summary/profile_text/memory_kind/status`。
6. 会话内记忆模块列表展示当前会话摘要记忆，支持手动修改 `summary/profile_text/memory_kind/status`。
7. 手动修改必须写入审计事件，且后续记忆注入以修改后的内容为准。

### 验收标准
- [ ] 用户可查看自己的画像记忆和会话记忆
- [ ] 用户可手动修改自己的用户画像记忆
- [ ] 用户可在会话内手动修改当前会话摘要记忆
- [ ] 用户可归档或删除不需要的记忆
- [ ] 管理员可筛选用户查看记忆
- [ ] 所有管理操作写入审计事件
