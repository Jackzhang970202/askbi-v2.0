# AskBI 统一会话与分析体验升级设计文档

## 1. 项目背景与升级目标

AskBI 当前已经具备数据库问数、Excel 分析、技能注入、智能体管理等能力，但用户使用路径、会话承载方式与分析结果展示方式仍存在割裂：普通对话、BI、Excel 与 Skill 增强态容易被理解为多个入口；历史记录与上下文恢复不稳定；图表结果呈现偏工程态，缺少产品化分析体验。

本次升级目标不是增加单点功能，而是将 AskBI 从“多个能力并列存在”升级为“统一会话承载多能力、统一历史恢复、统一分析结果交付”的产品形态。

| 升级方向 | 目标 | 业务价值 |
|------|------|------|
| 会话模型升级 | 所有能力统一挂载到同一会话 | 降低用户学习成本，减少入口分裂 |
| 路由方式升级 | 由当前会话上下文决定执行链路 | 提升系统一致性，减少误路由 |
| 能力管理升级 | Skill、智能体与记忆模块成为可配置能力体系 | 支撑后续场景化扩展 |
| 分析结果升级 | 图表结果从技术输出升级为产品化分析卡片 | 提升汇报展示效果与业务可读性 |


---

## 2. 产品形态总览

本次升级后的 AskBI 采用“统一聊天页 + Skill 能力挂载 + 记忆增强”的总体模型。

### 2.1 统一入口原则

用户进入系统后，以“新建对话”作为默认入口；所有后续能力，如数据库数据源、Excel 数据源、Skill 与记忆增强能力，均在当前会话内挂载或生效，不再要求为不同能力分别进入不同的独立会话模式。

### 2.2 统一会话承载能力

| 会话类型 | 定义 | 适用场景 | 切换方式 |
|------|------|------|------|
| 普通对话 | 未挂载任何执行上下文的默认会话 | 需求澄清、产品问答、通用问答 | 新建对话默认进入 |
| BI 会话 | 已挂载数据库数据源的会话 | 自然语言问数、图表分析 | 在当前会话中选择数据库数据源 |
| Excel 会话 | 已挂载 Excel 数据源或上传文件的会话 | 文件分析、表格洞察、图表生成 | 在当前会话中选择 Excel 数据源或上传文件 |
| Skill 增强会话 | 已选择一个或多个 Skill 的会话 | 专业任务执行、复杂问题解决、场景增强 | 在当前会话中选择 Skill |

### 2.3 统一结果交付原则

系统输出结构化结果组合，包括：

| 结果层 | 内容 | 展示方式 |
|------|------|------|
| 文本层 | 结论、分析说明、解读 | 消息正文 |
| 过程层 | 思考过程、执行阶段、流式进度 | 可折叠过程面板 |
| 数据层 | 表格、结果集、字段映射 | 表格视图 |
| 图表层 | 分析图表、图表配置、主题切换 | 图表卡片 + 编辑面板 |

## 3. 统一会话架构设计

### 3.1 会话核心模型

系统需要引入统一会话上下文模型，使每个会话都具备“当前承载能力”的明确状态。

| 字段 | 含义 | 说明 |
|------|------|------|
| 会话标识 | 当前会话唯一 ID | 同一会话内不因能力切换而变化 |
| 上下文类型 | 当前会话所属能力类型 | 至少支持普通对话、BI、Excel |
| 上下文引用 ID | 关联对象标识 | 例如数据源标识或能力配置标识 |
| 上下文引用名称 | 当前上下文名称 | 例如数据源显示名 |
| 数据源名称 | 当前问数或分析使用的数据源 | 作为 BI / Excel 执行依据 |
| Skill 列表 | 当前会话启用的 Skill 集合 | 作为能力增强与提示注入依据 |
| 记忆摘要 | 当前会话沉淀出的阶段性摘要 | 用于上下文压缩与恢复 |
| 用户画像记忆 | 用户长期偏好、角色、关注点 | 由 Mem0 持久化管理 |

### 3.2 会话状态流转

| 当前状态 | 用户操作 | 目标状态 | 说明 |
|------|------|------|------|
| 普通对话 | 选择数据库数据源 | BI | 后续消息进入数据库问数流程 |
| 普通对话 | 选择 Excel 数据源 | Excel | 后续消息进入 Excel 分析流程 |
| 普通对话 | 选择 Skill | Skill 增强 | 后续消息在当前能力基础上叠加 Skill 增强 |
| BI / Excel / Skill 增强 | 清除上下文 | 普通对话 | 历史保留，仅后续消息恢复为普通对话 |
| BI | 切换为 Excel | Excel | 不丢失历史，改为文件分析语义 |
| Excel | 切换为 BI | BI | 不丢失历史，改为数据库问数语义 |

### 3.3 历史记录设计

统一历史记录不是简单拼接多个列表，而是以“会话”为唯一对象展示历史，按上下文类型进行标签化区分。

| 展示维度 | 设计要求 |
|------|------|
| 标题 | 取首轮有效问题或默认标题，不依赖固定入口名称 |
| 标签 | 明确区分普通对话、BI、Excel 与 Skill 增强态 |
| 图标 | 不同上下文使用不同图标与颜色语义 |
| 恢复 | 点击任意历史会话后，必须恢复消息、上下文、输入区状态与执行路由 |
| 一致性 | 前端展示状态必须以后端会话状态为唯一事实来源 |

## 4. 前端交互与页面设计

### 4.1 统一聊天页结构

升级后的统一聊天页需要承载从会话创建、上下文挂载、消息交互到分析结果查看的完整闭环。

| 区域 | 功能 | 设计要求 |
|------|------|------|
| 左侧导航区 | 新建对话、配置页入口、历史记录 | 历史记录统一展示各类会话 |
| 顶部会话头部 | 当前会话标题、当前上下文标签、辅助操作 | 上下文标签可见且可清除 |
| 消息列表区 | 文本、思考过程、表格、图表结果 | 支持结构化分析结果展示 |
| 输入区 | 用户提问、能力挂载、技能选择、发送 | 作为统一交互主入口 |

### 4.2 输入区交互设计

输入区左侧提供“上下文挂载入口”，以菜单方式承载当前会话可切换的能力对象。

| 交互项 | 行为 | 结果 |
|------|------|------|
| 选择数据库数据源 | 将当前会话挂载到数据库上下文 | 后续消息走 BI 流程 |
| 选择 Excel 数据源 | 将当前会话挂载到 Excel 上下文 | 后续消息走 Excel 流程 |
| 选择 Skill | 为当前会话挂载一个或多个 Skill | 后续消息在当前能力基础上叠加 Skill 增强 |
| 清除上下文 | 解除当前能力挂载 | 后续消息回到普通对话 |
| 技能选择 | 指定当前会话可用 Skill 集合 | 影响本轮及后续调用能力 |

### 4.3 会话头部与标签设计

| 元素 | 设计要求 |
|------|------|
| 会话标题 | 体现当前对话主题，不与能力名混淆 |
| 上下文标签 | 显示当前数据源或 Skill 增强状态 |
| 标签颜色 | 与上下文类型对应 |
| 清除动作 | 标签就近提供清除入口 |
| 空态 | 无上下文时不显示能力标签，仅保留普通对话语义 |

### 4.4 记忆模块交互设计

本次升级需要引入基于 Mem0 的记忆模块，用于承接用户画像记忆与会话摘要记忆两类能力。

| 记忆类型 | 作用 | 生命周期 | 展示方式 |
|------|------|------|------|
| 用户画像记忆 | 记录用户角色、偏好、常用关注点、业务口径 | 跨会话长期保留 | 不直接暴露原始内容，可在系统配置中查看摘要 |
| 会话摘要记忆 | 记录当前会话阶段性结论、关键上下文、未完成事项 | 会话级持续更新 | 在对话执行时作为压缩上下文使用 |

## 5. 能力体系与后端架构设计

### 5.1 普通对话能力

系统需要新增普通对话默认能力，用于承接未挂载任何数据或 Skill 能力时的消息处理。

| 设计项 | 要求 |
|------|------|
| 能力角色 | 默认对话助手 |
| 触发条件 | 当前会话无任何执行上下文 |
| 输入 | 用户问题、上下文记忆、可选 Skill |
| 输出 | 自然语言答复 |
| 边界 | 不生成伪 SQL、不冒充数据分析结果 |

### 5.2 Skill 与智能体管理能力

本次升级需要将 Skill 和智能体管理纳入产品级配置体系，而不是仅作为底层实现概念。

| 模块 | 职责 | 设计目标 |
|------|------|------|
| Skill 管理 | 维护可注入能力片段 | 支持能力组合与会话增强 |
| 智能体管理 | 维护不同角色智能体 | 支持普通对话、BI、Excel 的角色扩展 |
| 绑定关系 | 管理 Skill 与智能体的关联 | 支撑按能力场景动态注入 |

### 5.3 记忆模块能力

记忆模块采用 Mem0 作为统一记忆底座，分为用户画像记忆与会话摘要记忆两层。

| 记忆层 | 职责 | 设计要求 |
|------|------|------|
| 用户画像记忆 | 沉淀长期偏好、角色信息、关注主题 | 跨会话持久化，按用户维度读取 |
| 会话摘要记忆 | 沉淀当前会话关键事实、阶段结论、上下文压缩结果 | 按会话维度更新与恢复 |
| 更新机制 | 在关键轮次后自动更新记忆 | 保持长期与短期记忆新鲜度 |

## 6. 图表分析体验设计

### 6.1 设计目标

图表结果不再只是“消息中的一个渲染块”，而是作为分析结论的重要承载体进行产品化设计。目标是让图表具备展示价值、汇报价值和二次调整能力。

### 6.2 图表结果卡片设计

| 区域 | 内容 | 设计要求 |
|------|------|------|
| 图表展示区 | 图表主体、必要图例 | 仅承载核心数据可视化，不放冗余控件 |
| 底部信息条 | 图表标题、图表类型图标、快捷操作 | 保持单行、简洁、易扫视 |
| 详情编辑区 | 图表/表格切换、样式和字段配置 | 提供完整的分析编辑能力 |
